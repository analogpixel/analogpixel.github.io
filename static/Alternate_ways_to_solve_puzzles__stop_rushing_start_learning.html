Alternate ways to solve puzzles
stop rushing start learning
1/9/2016

Project Euler is a website that hosts programming/math challenge that get progressively harder as you move from problem to problem. The most common way of solving these problems is to pick your favorite language, solve the problem, and then move on to the next problem; quickly trying to get to those wonderful virtual trophies.

But looking at the first problem:

If we list all the natural numbers below 10 that are multiples of 3 or 5, we get 3, 5, 6 and 9. The sum of these multiples is 23.

Find the sum of all the multiples of 3 or 5 below 1000. If you solved this with your favorite language of choice, what have you really learned? And what exactly is the big rush to move on to the next problem anyways?

The Alternative

Alternatively, you could find other ways to solve the problem; while it still helps to solve the problem in a traditional way first (or not); don't stop after your initial solving attempt.

  • Write a parser to solve the problem and learn how to write a parser.
  • Write a custom Gui to solve the problem and work on UI design; or learn how D3.js works.
  • Use the pattern to create a design and work on UI design; or learn how D3.js works.
  • Design a physical interface to solve the problem and work on mechanical design along with interacting a physical design with a computer.
  • What does a drum rhythm with different beats on multiples of 3, 5, and both send like? While it doesn't solve the problem, it does spring board you off into other directions of things to explore.
  • Learn verilog and create a circuit to solve the problem. Attack the problem with this mindset; You are working for a company, and this "problem" is the core work the factory workers need to do every day. Your job is to design an interface to allow them to easily solve this type of problem and all you have to work with is a touch screen and a cpu (the rest needs to be designed by you.)

Not only does solving the problem in this way give you a much deeper understanding of the problem, It also pushes you to learn new technologies and ,somewhat, practical uses for applying these new technologies.

The Parser

I used http://pegjs.org/ to create the parser for my simple language, it has an online tester, and was pretty easy to get started with. My language definition ended up looking like this:

start
  = goals+

goals
  = left:multset " I " right:goals "\n"* { return left.intersect(right); }
  / left:multset " U " right:goals "\n"* { return left.union(right);  }
  / multset

multset
  = "M:" a:int ":" b:int ":" c:int { return multSet(a,b,c); }

int
  = digits:[0-9]+ { return parseInt(digits.join(""),10); }

After using the pegjs website with the above, I got a parser I was able to place in my program. I can now solve the initial problem with:

par.parse("M:3:0:10 U M:5:0:10")[0].reduce( function (a,b) { return a+b; })

Custom D3.js gui for solving Project Euler Problem 1

This is still a work in progress.

Currently I have circles on the screen that represent the ranges of multiples. You can drag the circles around, and you can combine them by dragging them on-top of each other. The number at the bottom indicates the sum of all the numbers in the ranges that are contained in that circle. As an added bonus, if you double click on a circle that has more than one value in it, the last value added is popped off into a new circle.
